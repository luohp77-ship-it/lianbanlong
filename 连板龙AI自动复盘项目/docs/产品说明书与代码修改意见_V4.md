### B. 产品说明书修改项详细评价

#### B.1 收费模型（按年订阅制） — ✅ 合理，补充一个建议

**方向正确**，两点增补建议：

1. **2台设备 → 建议改为3台。** A 股用户通常有「台式机 + 笔记本 + 备用机」习惯，2台容易触发客服投诉，3台是合理的心理阈值。

2. **到期降级机制有实现陷阱。** 当前代码全部跑在**本地定时任务**上，没有服务端控制。到期后要「暂停每日更新」需要授权系统能**阻止 `--update` 命令行执行**。设计方案需明确：

```
方案A（严格）：license 验证不过 → --update 直接退出 → TDX数据停在最后一天
方案B（温和）：过期后降级为试用版，--update 仍可跑，但输出水印或限制功能
```

**建议选方案B。** 原因是定时任务在后台静默运行，如果用户过期后不知情，TDX 一直不更新，用户会以为是软件坏了而不是到期了，容易引发差评。

#### B.2 更新流程（双轮→单次18:30） — ✅ 与当前代码已对齐

完全符合当前代码现状。上一轮对话中已将定时任务改为**每日18:00单次**，并更新了 `设置定时更新.bat`。

**补充说明**：实际龙虎榜数据 **18:00-20:00间陆续公布**，18:30跑的话部分日期可能拿不全。建议说明书写 **18:30-19:00** 区间而非固定时间点。

#### B.3 删除微信推送 — ✅ 必须做，但注意代码里真的在调用

V4 草案认为微信推送是死代码，**实际情况是它在工作**：

| 位置 | 代码 | 状态 |
|------|------|------|
| `engine.py L490-507` | `_push_wechat()` 函数定义 | ✅ 完整实现 |
| `engine.py L946-947` | `_push_wechat()` 调用 | ✅ 每次更新都会调 |

- DEFAULT_CONFIG 同步清理

#### B.4 安装流程（拖拽+固定路径） — ⚠️ 优先级偏高，建议降为中

在当前阶段，安装流程重写的**优先级应该降到「中」**，理由是：

1. **当前安装是可用的。** `install.py` 能自动扫描注册表和常见路径找到通达信，失败才回退到 `input()`
2. **用户群体特征。** 会用「连板龙」的基本都是有经验的股民，手动选路径不是门槛
3. **拖拽会引入新依赖。** `tkdnd` 库的兼容性和打包复杂度需要额外测试，增加交付风险

**关于固定路径安装的技术说明：**

V4 草案说「exe 在临时目录」，这是 PyInstaller `--onefile` 模式的固有行为：
- 运行时解压到 `%TEMP%\_MEIxxxxx\`
- `sys.executable` 指向临时路径
- 定时任务如果指向临时路径，下次重启就失效

**正确做法不是改打包脚本**，而是**安装时复制 exe 到固定目录**（如 `%LOCALAPPDATA%\连板龙\连板龙V3.exe`），然后基于固定路径创建快捷方式和定时任务。这个改动可以在不重写安装流程的前提下独立完成。

**授权验证必须在以下所有入口生效：**

| 入口 | 场景 | 实现方式 |
|------|------|---------|
| GUI 启动 (`python app.py`) | 用户双击 exe | 弹窗/直接进 |
| `--update` 命令行 | 定时任务调用 | 不弹窗，验证失败后静默退出或降级 |

**license.py 需要覆盖的能力：**

```python
class LicenseManager:
    def activate(code, device_fingerprint)  # 联网激活 → 存 license.dat
    def verify()  # 本地验证（签名+时间+设备指纹）
    def bind_device(device_name)  # 绑定
    def unbind_device(device_fingerprint)  # 解绑
    def is_trial()  # 是否为试用期
    def days_remaining()  # 还剩多少天（用于弹窗提醒）
    def is_expired()  # 已过期 → --update 模式降级
```

**设备指纹建议**：MAC + 主机名 + C 盘序列号三者哈希，不要依赖 `wmic`（部分精简版系统没有）。

#### F.2 必须修复的问题（共 3 个）

**🔴 #1 license.py _parse_activation_code 未做签名验证**

当前的激活流程：用户输入 `LICENSE-XXXX-...` → `_parse_activation_code` 检查格式 → **直接返回 365 天授权**，没有任何 RSA 签名验证。

这意味着任何格式正确的激活码都能通过。需改为：
```python
def _parse_activation_code(self, code):
    # 格式检查后，用 _EMBEDDED_PUBLIC_KEY 验证签名
    # 只有 verify_signature(payload, signature) 为 True 才返回有效
```

修复后文件：`license.py` L304-342

**🔴 #2 generate_license.py --gen-code 与 license.py 密钥不配对**

`--gen-code` 无参数时调用 `gen_keys()` 生成**新密钥对**并用新私钥签名。但 `license.py` 里的 `_EMBEDDED_PUBLIC_KEY` 是旧的。签名无法通过验证。

修复方向：使用流程改为
1. `--gen-keys` 生成密钥对 → 将公钥手动更新到 license.py
2. 用生成的私钥 `--gen-code --key <私钥文件>`

**🔴 #3 （同 #1）此为当前最严重的安全漏洞，授权系统名字实亡**

授权系统的核心是 R1（签名验证），不修复等于授权系统没做。
