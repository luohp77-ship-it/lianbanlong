## 一、严重漏洞 / 架构缺陷

### 1. 激活码安全形同虚设

策划书写"客户端无法伪造校验码"——实际上本地验证只是个安慰。

- 截止日期明文编码在激活码中（`FP-20270601-...`），改日期就能绕过
- 校验码用对称密钥签名，密钥存在 `engine.exe` 里，反编译就能提取
- 没有服务端 = 没有吊销机制，一个激活码被公开后无法封禁
- "更换密钥"策略意味着需要强制所有老用户重新下载，跟换激活码没区别

**后果**：破解版最快几天就会出现。

**最低可行改进**：至少把校验逻辑放在多个分散的函数里，增加逆向难度。或者考虑用在线验证（每次启动只发一次心跳）。

### 2. 写了"零服务端"又需要 HTTP 请求

- 版本更新检查（6.3 节）需要访问网页获取 `latest_version`
- 龙虎榜依赖 ClawHub（`fffy520.gicp.net`）
- 同花顺 `data.10jqka.com.cn` 是外部 API

这些都需要网络。既然必须联网，不如直接在激活验证时加一次在线校验（一条带激活码哈希的 POST），成本极低。

### 3. 四个外部数据源全部是单点故障

| 数据源 | 故障后果 | 备选方案 |
|--------|----------|----------|
| 同花顺API | 涨停池/概念为空 | 写了东方财富备选 |
| zzshare | 概念热度完全丢失 | 写了"无" |
| ClawHub | 龙虎榜为空 | "暂时为空" |
| 东方财富 push2/push2ex | 封板/炸板/跌停全丢 | 写了"无" |

对于 zzshare 和东方财富的备选方案，策划书写了"无"。

- **zzshare**是概念排名的唯一来源。如果它挂了，`概念主线` / `概念次主线` / `潜在概念`三个板块全部报废。
- 东方财富 `push2ex` 是封板/炸板事件唯一来源，直接影响 `DT(今跌停)` `ZBQ(曾涨停)` `ZDT(曾跌停)` 三个板块。

### 4. 没有数据一致性校验逻辑

策划书 4.1 说"zzshare + 东方财富成分股双重验证"，但：

- 两个源数据格式不同，怎么对齐？
- 冲突时以谁为准？
- 同花顺涨停池和东方财富涨停池可能不一致（**涨停定义不同**），按谁来？
- 4.1 连板算法只说了"前5个交易日数据"，但没有说明数据源取哪个

### 5. 写入文件没有原子性保证

引擎直接覆盖 `.blk` / `blocknew.cfg` / `gridtab.dat`：

- 如果写入中途崩溃，交易软件读到半截文件会导致**标签栏乱码甚至闪退**
- 没有旧文件备份，出问题无法回滚
- 建议：先写临时文件，完整写入后再 `rename` 替换

### 6. 连板天数只回溯5天不够

```
for each 历史交易日: if 该股在历史日也涨停: 天数 += 1
```

- 假设某个妖股已经15连板，回溯5天只算出"5天5板"
- 会出现 **"15连板被错标为5板"** 的致命Bug
- 应该从 `昨日涨停数据` 里的连板数 + 1，而不是从0开始追溯

### 7. 安装权限问题

策划书说"双击 install.exe → 写入板块配置"。但：

- 通达信如果装在 `C:\Program Files` 下，普通用户没权限写 `T0002/blocknew/`
- `blocknew.cfg` 和 `gridtab.dat` 的二进制格式没有文档，靠逆向工程写的格式可能在下个版本被改
- 券商定制版通达信的文件路径、文件格式可能完全不同
- 没有提"权限不足时怎么办"

## 二、产品逻辑缺陷

### 8. 概念板块"互斥不重叠"规则有问题

```
主线 = 第1名
次主线 = 第2名（排除已入主线的股票）
潜在 = 第3名（排除已入主线+次主线的股票）
```

但实际情况是一只股票**同时属于多个概念**：

- 比如某半导体股，同时属于"半导体"和"人工智能"
- 如果"半导体"排第1、股票A已被归入主线，那么即使A也属于"人工智能"（排第2），它也不会出现在次主线里
- **次主线的涨停数量就被低估了**

正确做法应该是：概念排名按"该概念下所有涨停股"排名（不管股票是否已被其他概念占用），板块展示按互斥。目前策划书算的是板块展示互斥，这个是对的。但排名要区分"概念涨停力"和"概念展示股"。当前方案排名也是互斥的，会漏。

### 9. 概念映射表太简略且静态

- 一共只有约25条映射，每天市场出现的概念远不止这些
- "不在映射表中的直接用原始名"——原始名可能是 `zzshare` 的内部ID，对用户没有意义
- 映射表硬编码在程序里，新概念出现时无法更新（要重新发布版本？）

### 10. 时间线 Bug

```
16:00 定时任务触发 → 获取涨停数据
```

- 收盘是15:00，但不是所有股票都在15:00前完成竞价
- 特别是科创板、北交所，收盘后还有收盘集合竞价（15:05-15:30）
- 龙虎榜数据通常18:00后才出
- 16:00获取的数据**大概率不完整**

建议：定时任务设多个时间点，或首次16:00获取涨停/跌停/概念，18:30补充龙虎榜。

## 三、工程实现缺失

### 11. 完全没有提异常处理和日志

- API超时怎么办？重试几次？
- API返回空数据怎么办？是挂着昨天的数据还是清空？
- 网络断了怎么办？
- 引擎运行中有任何错误弹窗通知用户吗？
- 策划书提到 `logs/` 目录但没说日志格式、日志级别、自动清理策略

### 12. 没有测试策略

- "同花顺版预计1天"——1天能测试多少边界情况？
- 不同券商通达信的文件路径不同，需要分别测试
- 交易日/非交易日切换测试
- ST股票涨停5%边界计算测试
- 跨年/跨月日期计算测试

### 13. 没有性能考量

策划书说"for each 涨停股票 → for each 所属概念"，今天全市场涨停数量通常100-300只，每只可能跨5-8个概念，循环次数不大。但如果未来扩展到全市场概念扫描或历史回溯，没有提到性能边界。

### 14. 版本更新逻辑有漏洞

```
读取本地版本号 → 访问网页获取最新版本号 → 版本号落后 → 对话框提示
```

- 用户点了"稍后更新"怎么办？有没有强制更新策略？
- 如果服务端下线，所有用户得到的是"无法检查更新"还是报错？
- 旧版本和新版本的 `.blk` 格式可能不兼容，混用会出问题

### 15. Launcher 逻辑不完整

```
检查是否今天已更新 → 否 → 运行引擎 → 更新数据 → 启动交易软件
```

- "运行引擎"是同步等待还是异步？如果引擎卡死10分钟用户等着？
- 引擎失败时（API挂了），是跳过自己打开软件还是报错退出？
- 有多快启动交易软件？用户可能以为程序死了

## 四、商业/合规风险

### 16. 同花顺/东方财富 API 使用风险

- 对外销售99元/年的商业产品，**未经授权爬取第三方API数据**，法律风险高
- 同花顺和东方财富都有反爬机制
- 数据量大了之后可能被封IP
- 建议加免责声明说明数据来自公开接口，或申请商业数据接口

### 17. 淘宝销售合规

- 虚拟商品需要淘宝**营业执照**
- "炒股工具"类的商品可能触发淘宝风控（涉及投资建议？）
- 商品描述如果包含"涨停板""龙头战法"等词，可能被平台判定为荐股内容下架

### 18. 没有竞品分析

目前淘宝上已有大量"涨停板复盘表格"类产品，差异化在哪？策划书没有提到。

## 五、快速改进建议优先级

| 优先级 | 问题 | 改进 |
|--------|------|------|
| P0 | 连板天数只回溯5天 | 改为从昨日数据继承+1 |
| P0 | 文件写入无原子性 | 先写临时文件再rename |
| P0 | 16:00数据不完整 | 分两次定时（16:00+18:30） |
| P0 | zzshare唯一源无备选 | 至少加一个东方财富概念排名备用 |
| P1 | 安装权限问题 | install.exe需要管理员提权或检测路径权限 |
| P1 | API异常处理缺失 | 加3次重试+降级策略+默认数据策略 |
| P1 | 激活码安全薄弱 | 至少加在线验证一次 |
| P2 | 概念映射表静态 | 改为云端配置文件，程序启动时拉取 |
| P2 | 异常通知缺失 | 引擎失败时弹窗或在交易软件内提示 |
| P3 | 日志简陋 | 加分级日志、自动清理、错误摘要 |
