## 二、致命问题（必须修复，否则无法正常运行）

### ❌ 2.1 概念分析模块调用参数不匹配（运行时 TypeError）

**位置**：`engine.py` 第 579 行 → `concept_analyzer.py` 第 193 行

**代码**：
```python
# engine.py:579  →  调用 analyze_concepts 时传入 up_stocks 参数
concept_result = analyze_concepts(limit_up_codes, callback=log, up_stocks=up_stocks)

# concept_analyzer.py:193  →  函数签名不接受 up_stocks 参数
def analyze_concepts(limit_up_codes, callback=None):
```

**问题**：
1. `engine.py` 调用 `analyze_concepts()` 时传入了 `up_stocks=up_stocks`，但 `concept_analyzer.py` 中的函数定义 **不接受** `up_stocks` 参数
2. 运行时将抛出 `TypeError: analyze_concepts() got an unexpected keyword argument 'up_stocks'`，**导致整个连板龙引擎崩溃**
3. 产品说明书宣称"V2.1 概念分析改用 reason_type 分组"，但 `concept_analyzer.py` 中真正实现 reason_type 分组的函数是 `analyze_concepts_by_reason()`，从未被 `engine.py` 调用
4. `analyze_concepts()` 仍使用已失效的 push2 API（返回空数据），实际概念分析在 V2.1 中**完全断链**

**影响**：
- 概念热点板块（RDGN/HYRD/RDGN3）输出为空
- `extern_user.txt` 中的"所属概念"列为空
- 用户看到的概念分析功能完全不可用

**修复建议**（择一）：

**方案 A（推荐，最小改动）**：
```python
# engine.py 修改：
# 将第 579 行改为：
concept_result = analyze_concepts_by_reason(up_stocks, callback=log)
```

**方案 B（接口改造）**：
```python
# concept_analyzer.py 修改 analyze_concepts 函数，增加 up_stocks 参数并内部调用 analyze_concepts_by_reason：
def analyze_concepts(limit_up_codes, callback=None, up_stocks=None):
    if up_stocks:
        return analyze_concepts_by_reason(up_stocks, callback=callback)
    # fallback: push2 API
    ...
```

**风险评估**：🔴 致命 — 运行时必然崩溃，概念分析 100% 不可用

| `app.py` 第 41 行 | `V5` | 窗口标题 `self.root.title('连板龙 V5')` |
| `engine.py` 第 3 行 | `V2.1` | 文档字符串 "连板龙引擎 V2.1" |
| `tg_bot.py` 第 3 行 | `V5` | 文档字符串 "连板龙 V5 Telegram Bot" |

### ⚠️ 3.2 跌停板块定义与说明书不一致

**位置**：`engine.py` 第 660-661 行 vs 产品说明书表 2.1

| 来源 | DT 板块定义 |
|:-----|:-----------|
| 产品说明书表 2.1 | `DT = 今跌停 = 封跌停+曾跌停` |
| 产品说明书表 2.1 第 58 行 | `DT 内容：封跌停+曾跌停` |
| `engine.py` 第 212 行代码注释 | `all_dting = 封跌停 + 曾跌停（所有今日触及跌停的股票）` |
| `engine.py` 第 660 行 | `_write_blk_verified(..., 'DT(全部跌停)', ...)` ✅ 与代码一致 |

经过核对，**代码实现与说明书一致**，但板块标签命名稍有不统一：blk 文件写的是 `今跌停`（gridtab.dat），而代码日志写的是 `DT(全部跌停)`。用户看到通达信标签是"今跌停"，数据包含所有触及跌停股。**无运行时问题，但建议说明书中更明确标注"含曾跌停"。**

---

### ⚠️ 3.3 TDX_DIR 全局变量非线程安全

**位置**：`utils.py` 第 332-342 行

```python
TDX_DIR = None  # 模块级全局变量

def set_tdx_dir(tdx_path):
    global TDX_DIR
    TDX_DIR = tdx_path
```

**问题**：`engine.py` 在 `run_engine()` 中调用 `set_tdx_dir(tdx_dir)` 设置全局变量，然后 `calc_board_days_tdx()` → `read_tdx_daily()` 读取该变量。如果两个线程同时调用 `run_engine()`（理论上 GUI 的"立即更新"按钮可能被重复点击），`TDX_DIR` 可能被覆写为错误值。

**当前影响**：`app.py` 有 `self.running` 互斥锁防止重复点击，单线程模式下问题不大。但命令行模式 (`--update`) 没有此保护。

**修复建议**：将 `TDX_DIR` 改为线程局部存储（`threading.local()`），或作为参数直接传递给 `read_tdx_daily()` 和 `calc_board_days_tdx()`。

---

### ⚠️ 3.4 config.json 中 tgBotToken 明文存储

**位置**：`app.py` 第 260-262 行（设置面板中等号框用 `show='*'` 掩码但保存时明文写入 config.json）

**问题**：TG Bot Token 以明文 JSON 存储在本地文件。如果用户共享 config.json 或上传到网盘，Token 泄露后攻击者可通过 Bot 向用户发送恶意消息。

**修复建议**：用 `keyring` 库存储敏感凭证，或至少做简单的 Base64 编码（虽非真正加密，但可防肉眼泄密）。

## 四、中等问题（影响用户体感，建议下版本修复）

### 🔸 4.1 首次安装后概念板块为空

**现象**：用户双击 `install.exe` → 安装完成后运行首次数据更新 → 概念热点板块（RDGN/HYRD/RDGN3）为 **0 只股票**。

**根因**：见 2.1 致命 Bug。即使修复后，概念分析依赖同花顺返回的 `reason_type` 字段。如果某日该字段为空（同花顺 API 变更），概念板块同样为空。

**建议**：在概念分析返回空时，降级为简单的"按行业自动归类"（从同花顺的 `industry` 字段分组），至少给用户一个有意义的默认输出。

---

### 🔸 4.2 节假日文件缺失无默认兜底

**位置**：`utils.py` 第 448-456 行

```python
def _load_holidays():
    fp = STANDALONE_DIR / 'holidays.json'
    data = load_json(fp)
    _holidays_cache = set(data) if data else set()
    return _holidays_cache
```

**问题**：如果 `standalone/holidays.json` 缺失且非周末非假日运行，`is_trading_day()` 会误判交易日。在长假期（如春节）期间，定时任务可能在非交易日运行，触发 API 请求获取空数据。

**建议**：内置 2026 年关键假日兜底列表，或从在线日历 API 获取。

---

### 🔸 4.3 概念映射表（concept_map.json）过于简略

**位置**：`standalone/concept_map.json`

**问题**：产品说明书 V1 审核意见（#9）指出"映射表太简略且静态"。当前代码采用 `load_concept_map()` + `_cmap.get(name, name)` 策略（用原始名作为兜底），在 V2.1 中已有所缓解。但新概念出现时仍需手动更新映射表，否则用户看到的是原始 reason_type 名称（可能包含内部 ID）。

---

### 🔸 4.4 install.py 中定时任务的 PowerShell 脚本有编码风险

**位置**：`install.py` 第 104-108 行

```python
ps_script = f"""
$action = New-ScheduledTaskAction -Execute "{python_exe}" -Argument "\\"{app_py}\\" --update"
...
"""

ps_file = os.path.join(os.environ["TEMP"], "fupan_create_task.ps1")
with open(ps_file, "w", encoding="utf-8") as f:
    f.write(ps_script)
```

**问题**：
1. `python_exe` 和 `app_py` 路径中可能包含空格和特殊字符，需要额外转义
2. 如果 `python_exe` 路径中含中文字符（如 `C:\用户\...`），PowerShell 可能解析失败
3. 创建定时任务需要管理员权限，部分用户可能遇到权限错误

---

### 🔸 4.5 TG Bot 广播无速率限制

**位置**：`engine.py` 第 365-395 行

**问题**：`_push_tg()` 函数遍历所有用户并逐一发送消息，没有调用间延迟或失败重试。如果用户数量较大且 Telegram API 限流（通常 30 条/秒），后续消息会失败。

**建议**：加入 `time.sleep(0.05)` 调用间延迟，并为失败的消息记录到日志。

## 五、低优先度问题（改善项，不阻塞交付）

### 💡 5.1 自定义数据列（extern_user.txt）需用户手动配置

**位置**：产品说明书 2.3 章节

产品说明书写"用户需手动配置：功能→公式系统→自定义数据管理器→新建数据号1/2"。这是整个产品中**唯一需要用户手动操作的步骤**，对不熟悉通达信菜单的用户构成体验障碍。

**建议**：在安装完成提示中增加截图指引，或尝试通过修改通达信的配置文件（如 `attrib.ini`）自动创建数据号。

---

### 💡 5.2 日志文件没有大小限制

**位置**：`utils.py` 第 98-131 行 (`Logger.emit()`)

**问题**：日志每秒追加到 `logs/YYYYMMDD.log`，每天一个文件。`clean_old_logs(days=30)` 只按时间清理。如果一个交易日 API 故障反复重试，单日日志文件可能增长到几十MB。

**建议**：增加单文件大小限制（如 10MB 后轮转）。

---

### 💡 5.3 ZFB2 板块（昨涨停今）依赖 boards 缓存可能为空

**位置**：`engine.py` 第 676-689 行

**问题**：新安装用户首次运行时，`boards` 缓存为空，ZFB2 依赖同花顺历史 API 降级获取昨日涨停。如果同花顺 API 也失败，ZFB2 板块为空。用户看到"昨涨停今"板块无数据会困惑。

**建议**：无数据时在板块中显示原因提示（虽然通达信不支持文字提示，但可在日志中说明）。
